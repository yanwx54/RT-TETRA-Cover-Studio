## 1. 文档目的

本文档用于约束人类开发者、Codex 和其他 AI Agent 在本项目中的协作方式。

进入开发任务前，应优先阅读：

1. `PROJECT_INDEX.md`
2. `README.md`
3. `docs/01_产品开发项目需求书.md`
4. `docs/02_系统总体设计.md`
5. 当前任务相关专题文档

---

## 2. 总体原则

- 先确认当前阶段，再开始修改；
- 只修改与任务直接相关的文件；
- 不为未来可能需求提前设计复杂架构；
- 不在 GUI 层实现核心计算；
- 修改计算公式必须同步更新测试和文档；
- 发现需求与设计冲突时，先暂停并说明冲突。

---

## 3. 编码前检查

执行任何编码任务前，应检查：

- 当前 Git 状态；
- `PROJECT_INDEX.md` 中的当前阶段；
- 相关文档是否已有明确约束；
- 是否存在未提交的用户变更；
- 本次任务成功标准是什么。

如果任务是修复 Bug，应先写能复现问题的测试。

如果任务是新增校验，应先定义异常输入测试。

---

## 4. 修改范围规则

允许：

- 修改任务直接相关文件；
- 删除本次修改产生的无效导入和废弃变量；
- 为新增功能增加必要测试；
- 同步更新相关文档。

不允许：

- 顺手重构无关模块；
- 擅自删除原有文档或代码；
- 擅自改变目录结构；
- 将临时报告、缓存、日志提交；
- 在未确认技术栈前引入大量依赖。

---

## 5. 计算模块规则

核心计算必须满足：

- 可脱离 GUI 单独测试；
- 输入和输出单位明确；
- 公式来源可追溯；
- 中间计算过程可记录；
- 浮点比较使用容差；
- 异常输入必须明确报错。

禁止：

- 在按钮事件里直接写公式；
- 报告导出时重新计算；
- 模型内部扣除工程裕度；
- 静默吞掉计算异常。

---

## 6. 文档同步规则

以下变化必须更新文档：

- 新增或修改传播模型；
- 新增或修改输入参数；
- 修改计算公式；
- 修改目录结构；
- 修改 V1.0 范围；
- 新增导出格式；
- 新增运行命令。

优先更新：

- `PROJECT_INDEX.md`
- `README.md`
- 对应 `docs/` 专题文档

---

## 7. 测试规则

新增核心逻辑必须有测试。

最低要求：

- 正常输入；
- 边界输入；
- 非法输入；
- 计算结果容差；
- 异常路径。

测试应放在 `tests/`，不依赖手动 GUI 操作。

---

## 8. Git 规则

- 提交前检查 `git status`；
- 提交信息应说明修改目的；
- 不提交虚拟环境、缓存、生成报告；
- 推送前确认远程分支；
- 未经明确要求，不执行破坏性 Git 操作。

---

## 9. 推荐任务流程

```text
阅读 PROJECT_INDEX.md
↓
确认任务边界
↓
阅读相关文档
↓
制定最小修改方案
↓
实现或补文档
↓
运行验证
↓
更新 PROJECT_INDEX.md
↓
汇报变更和未完成项
```
